# 3.6 Supplier

`Supplier<T>` is one of Java's built-in Functional Interfaces.

It represents an operation that **does not accept any input** but **returns a value**.

Unlike `Consumer`, which performs an action, `Supplier` generates or supplies data whenever it is requested.

It is commonly used for **object creation**, **lazy initialization**, **configuration loading**, **random value generation**, and **factory methods**.

---

# Why was Supplier introduced?

Many applications need to generate or retrieve values without requiring any input.

Examples include:

- Generate a random number.
- Create a new object.
- Get the current date and time.
- Read a configuration value.
- Fetch a database connection.

Before Java 8, these operations were usually implemented using separate methods.

Java 8 introduced `Supplier` so that value generation could be treated as reusable behavior and passed wherever needed.

---

# What Problem Does Supplier Solve?

Suppose an application needs a new `Employee` object.

Instead of creating the object directly every time,

```java
Employee employee = new Employee();
```

we can define how an Employee should be created.

```java
() -> new Employee()
```

Whenever the application needs an Employee, it simply asks the Supplier.

The Supplier generates the value when requested.

# Definition

`Supplier<T>` is a built-in Functional Interface available in the `java.util.function` package.

It accepts no input and returns one value.

---

# Method Signature

```java
@FunctionalInterface
public interface Supplier<T> {

    T get();

}
```

```java

↓

Supplier

↓

T
```

Meaning

```java

↓

↓

Output
```

# The get() Method

The `get()` method generates or retrieves a value.

Syntax

```java
T get();
```

Example

```java
Supplier<String> greeting =
        () -> "Hello Java";

System.out.println(greeting.get());
```

Output

```java
Hello Java
```

# Built-in Methods

Unlike `Predicate`, `Function`, and `Consumer`, `Supplier` does **not** provide any additional default methods.

It contains only one abstract method.

```java
T get();
```

Its simplicity makes it one of the easiest Functional Interfaces to understand.

---

# Real-world Usage

`Supplier` is widely used whenever an application needs to **generate or retrieve a value** without requiring any input.

Common production use cases include:

- Object creation
- Factory methods
- Lazy initialization
- Stream API (`Stream.generate()`)
- Random value generation
- Reading configuration values
- Getting the current date and time
- Dependency Injection frameworks such as Spring

---

## Stream API

`Stream.generate()` expects a `Supplier`.

```java
Stream.generate(() -> "Java")
      .limit(5)
      .forEach(System.out::println);
```

Output

```
Java
Java
Java
Java
Java
```

Here,

- `generate()` expects a `Supplier`
- Every call to `get()` produces the next element of the Stream

---

## Object Creation

```java
Supplier<Employee> employeeSupplier =
        Employee::new;

Employee employee = employeeSupplier.get();
```

Instead of creating objects directly, the creation logic can be encapsulated inside a Supplier.

---

## Lazy Initialization

Sometimes creating an object is expensive.

Instead of creating it immediately, we delay creation until it is actually needed.

```java
Supplier<DatabaseConnection> connectionSupplier =
        DatabaseConnection::new;
```

The object is created only when:

```java
connectionSupplier.get();
```

is called.

---

## Generating Random Values

```java
Supplier<Integer> random =
        () -> new Random().nextInt(100);

System.out.println(random.get());
```

Each invocation generates a new random number.

---

## Current Date and Time

```java
Supplier<LocalDateTime> now =
        LocalDateTime::now;

System.out.println(now.get());
```

The Supplier always returns the current date and time when requested.

---

# Best Practices

- Use `Supplier` when no input is required.
- Prefer Method References for object creation.
- Keep Suppliers focused on generating a single value.
- Avoid performing heavy business logic inside `get()`.
- Use Suppliers for lazy initialization whenever appropriate.

---

# Common Mistakes

- Using `Supplier` when input is required (use `Function` instead).
- Assuming `get()` always returns a new object.
- Writing complex business logic inside a Supplier.
- Forgetting that different implementations may return different values on each call.

---

# Common Interview Traps

- `Supplier` accepts **no input**.
- `get()` is the Single Abstract Method (SAM).
- `Supplier` returns one value.
- `Stream.generate()` expects a `Supplier`.
- `Supplier` is commonly used for lazy object creation.

# Interview Scenario / Real-world Motivation

**Scenario: Expensive Object Creation**

Suppose creating a database connection takes time and resources.

Instead of creating it during application startup, we delay its creation until it is actually needed.

```java
Supplier<DatabaseConnection> connectionSupplier =
        DatabaseConnection::new;
```

Only when the application requires the connection do we execute:

```java
DatabaseConnection connection =
        connectionSupplier.get();
```

This approach reduces unnecessary resource usage and improves application startup time.

This pattern is widely used in enterprise applications and dependency injection frameworks.
