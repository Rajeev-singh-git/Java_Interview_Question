# 4. Method & Constructor References

Method References are one of the most useful features introduced in **Java 8**.

They provide a **shorter and more readable way** to write certain Lambda Expressions.

A Method Reference **does not introduce any new functionality**. It is simply a concise way to refer to an existing method or constructor.

They are heavily used with **Streams**, **Collections**, **Comparator**, **CompletableFuture**, and many modern Java libraries.

---

# Why were Method References introduced?

Consider the following Lambda Expression.

```java
names.forEach(name -> System.out.println(name));
```

The Lambda simply calls an existing method.

Since no additional logic is being performed, writing the entire Lambda is unnecessary.

Java 8 allows us to replace it with a Method Reference.

```java
names.forEach(System.out::println);
```

The result is:

- Less code
- Better readability
- Cleaner functional-style programming

---

# What Problem Do Method References Solve?

Method References solve one simple problem:

> **"Why write a Lambda if all it does is call an existing method?"**

For example,

Instead of writing

```java
name -> name.toUpperCase()
```

we can simply write

```java
String::toUpperCase
```

Instead of writing

```java
() -> new Employee()
```

we can write

```java
Employee::new
```

The behavior remains exactly the same.

Only the syntax becomes shorter.

---

# Mental Model

Think of Method References as **shortcuts for simple Lambda Expressions**.

```java
 Lambda Expression
        │
        ▼
Calls Existing Method
        │
        ▼
    Replace with
  Method Reference
```

Examples

```java
name -> System.out.println(name)
```

↓

```java
System.out::println
```

---

```java
name -> name.toUpperCase()
```

↓

```java
String::toUpperCase
```

---

```java
() -> new Employee()
```

↓

```java
Employee::new
```

Whenever a Lambda only calls an existing method or constructor, think **Method Reference**.

---

# What is a Method Reference?

A Method Reference is a shorthand notation for a Lambda Expression that directly calls an existing method.

Instead of writing the implementation yourself, you simply reference the method using the `::` operator.

General Syntax

```java
ClassName::methodName
```

or

```java
objectReference::methodName
```

or

```java
ClassName::new
```

---

# Types of Method References

Java supports four types of Method References.

1. Static Method Reference
2. Instance Method Reference of a Particular Object
3. Instance Method Reference of an Arbitrary Object
4. Constructor Reference

---

## 1. Static Method Reference

Used when the Lambda calls a static method.

Lambda

```java
s -> Integer.parseInt(s)
```

Method Reference

```java
Integer::parseInt
```

Another example

Lambda

```java
(a, b) -> Integer.sum(a, b)
```

Method Reference

```java
Integer::sum
```

---

## 2. Instance Method Reference of a Particular Object

Used when the Lambda calls a method on a specific object.

```java
Printer printer = new Printer();
```

Lambda

```java
message -> printer.print(message)
```

Method Reference

```java
printer::print
```

Here,

`printer` is a specific object.

---

## 3. Instance Method Reference of an Arbitrary Object

Used when the method is called on the object supplied at runtime.

Lambda

```java
name -> name.toUpperCase()
```

Method Reference

```java
String::toUpperCase
```

Another example

Lambda

```java
name -> name.length()
```

Method Reference

```java
String::length
```

Although we write the class name (`String`), the actual object is supplied automatically when the method is invoked.

This is the type of Method Reference most commonly used with the Stream API.

---

## 4. Constructor Reference

Used when the Lambda creates a new object.

Lambda

```java
() -> new Employee()
```

Method Reference

```java
Employee::new
```

Parameterized constructor

Lambda

```java
name -> new Employee(name)
```

Method Reference

```java
Employee::new
```

The compiler automatically matches the constructor based on the Functional Interface.

---

# How Does the Compiler Know Which Method to Call?

This is where **Functional Interfaces** come into play.

Suppose we have:

```java
Function<String, Integer> length =
        String::length;
```

The compiler already knows that `Function` expects:

```java
R apply(T t);
```

Since:

- Input → `String`
- Output → `Integer`

the compiler matches it with:

```java
String.length()
```

Similarly,

```java
Supplier<Employee> supplier =
        Employee::new;
```

matches

```java
Employee()
```

because `Supplier` expects:

```java
Employee get();
```

The compiler uses the target Functional Interface to determine which method or constructor the Method Reference refers to.

---

# When Can a Lambda Be Replaced?

A Lambda can be replaced by a Method Reference **only if it simply calls an existing method or constructor without adding extra logic**.

✔ Can be replaced

```java
name -> name.toUpperCase()
```

↓

```java
String::toUpperCase
```

---

✔ Can be replaced

```java
x -> Math.abs(x)
```

↓

```java
Math::abs
```

---

❌ Cannot be replaced

```java
name -> name.trim().toUpperCase()
```

because multiple operations are being performed.

---

❌ Cannot be replaced

```java
name -> "Hello " + name
```

because additional logic exists.

---

> **Note:** Method References improve readability but do not provide any new functionality. Every Method Reference can be written as a Lambda Expression, but not every Lambda Expression can be converted into a Method Reference.

---

# Lambda vs Method Reference

| Lambda Expression               | Method Reference                 |
| ------------------------------- | -------------------------------- |
| Defines behavior                | Refers to an existing method     |
| Can contain custom logic        | Can only call an existing method |
| More flexible                   | More concise                     |
| Suitable for complex operations | Suitable for simple method calls |

Example

Lambda

```java
names.forEach(name -> System.out.println(name));
```

Method Reference

```java
names.forEach(System.out::println);
```

Both produce exactly the same result.

# Best Practices

- Prefer Method References when they improve readability.
- Keep using Lambda Expressions if additional logic is required.
- Use Constructor References instead of Lambdas for object creation.
- Do not replace every Lambda with a Method Reference—choose the version that is easier to understand.

---

# Common Mistakes

### Using Method References Everywhere

Method References should improve readability, not reduce it.

Choose the form that makes the code easier to understand.

---

### Assuming Method References are Faster

Method References do **not** provide any performance advantage.

They are simply a different syntax for Lambda Expressions.

---

### Trying to Convert Every Lambda

Not every Lambda Expression can be converted into a Method Reference.

For example,

```java
name -> name.trim().toUpperCase()
```

cannot be replaced because additional processing is involved.

---

# Interview Tips

- Method References are **syntactic sugar** for Lambda Expressions.
- Every Method Reference can be written as a Lambda.
- Not every Lambda can be converted into a Method Reference.
- Method References always require a Functional Interface.
- Constructor References are simply Method References that refer to constructors.
