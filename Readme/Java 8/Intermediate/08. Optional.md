# 08. Optional

`Optional` is a container object introduced in **Java 8** to represent the **presence or absence of a value**.

Its primary goal is to reduce the chances of encountering **`NullPointerException (NPE)`**, one of the most common runtime exceptions in Java applications.

Instead of returning `null`, a method can return an `Optional`, making the absence of a value explicit.

---

# Why was Optional Introduced?

Before Java 8, methods often returned `null` when a value was unavailable.

Example:

```java
public Employee findEmployee(int id) {

    if (employeeExists(id)) {
        return employee;
    }

    return null;
}
```

The caller must remember to perform a null check.

```java
Employee employee =
        findEmployee(101);

System.out.println(
        employee.getName()
);
```

If `findEmployee()` returns `null`, the application throws a:

```text
NullPointerException
```

---

# The Problem with null

Using `null` has several disadvantages:

- Easy to forget null checks.

- Runtime failures (`NullPointerException`).

- Makes APIs less expressive.

- Difficult to understand whether `null` is an expected result.

---

# Using Optional

Instead of returning `null`, return an `Optional`.

```java
public Optional<Employee> findEmployee(int id) {

    if (employeeExists(id)) {
        return Optional.of(employee);
    }

    return Optional.empty();
}
```

The caller immediately knows that the value **may or may not be present**.

---

# What is Optional?

Think of `Optional` as a **container**.

```text
Optional
     │
 ┌───┴────┐
 │        │
Value   Empty
```

Instead of asking:

> Is this object `null`?

we ask:

> Does this `Optional` contain a value?

---

# Creating an Optional

There are three common ways to create an `Optional`.

---

# Optional.of()

Use `Optional.of()` when the value is **guaranteed to be non-null**.

```java
Employee employee =
        new Employee("Raj");

Optional<Employee> optional =
        Optional.of(employee);
```

Passing `null` throws:

```java
Optional.of(null);
```

```text
NullPointerException
```

---

# Optional.ofNullable()

Use `Optional.ofNullable()` when the value **may be null**.

```java
Employee employee =
        findEmployee(101);

Optional<Employee> optional =
        Optional.ofNullable(employee);
```

If `employee` is:

- not null → Optional contains the value.

- null → Returns `Optional.empty()`.

This is the most commonly used factory method.

---

# Optional.empty()

Creates an empty `Optional`.

```java
Optional<Employee> optional =
        Optional.empty();
```

It explicitly represents:

> "No value is available."

instead of returning `null`.

---

# of() vs ofNullable() vs empty()

| Method         | Accepts null? | Result                          |
| -------------- | ------------- | ------------------------------- |
| `of()`         | ❌ No          | Exception if value is null      |
| `ofNullable()` | ✅ Yes         | Empty Optional if value is null |
| `empty()`      | —             | Always empty                    |

---

# isPresent()

Checks whether an `Optional` contains a value.

```java
Optional<String> language =
        Optional.of("Java");

if (language.isPresent()) {

    System.out.println(
            language.get()
    );
}
```

Output

```text
Java
```

> **Note:** Frequent use of `isPresent()` followed by `get()` is generally considered an anti-pattern because it resembles traditional null checking. We'll see better alternatives in Part 2.

---

# ifPresent()

Executes an action only if a value is present.

It expects a **Consumer**.

```java
Optional<String> language =
        Optional.of("Java");

language.ifPresent(System.out::println);
```

Programming model:

```text
ifPresent()
        │
expects
Consumer
        │
implemented using
Method Reference
or
Lambda Expression
```

Unlike `isPresent()`, there is no explicit conditional check.

---

# Real-world Usage

`Optional` is commonly used as the return type of methods where a value may or may not exist.

Examples:

- Find Employee by ID

- Find User by Email

- Find Product by ID

- Find Order by Order Number

Instead of returning `null`, the method returns an `Optional`.

---

# Key Points to Remember

- `Optional` represents the presence or absence of a value.

- It helps reduce `NullPointerException`.

- Prefer `Optional.ofNullable()` when a value may be `null`.

- Use `Optional.empty()` instead of returning `null`.

- Prefer `ifPresent()` over `isPresent()` whenever appropriate.

---

# orElse()

`orElse()` returns the value if it is present; otherwise, it returns a default value.

```java
Optional<String> language =
        Optional.empty();

String result =
        language.orElse("Java");

System.out.println(result);
```

Output

```text
Java
```

Use `orElse()` when the default value is readily available.

---

# orElseGet()

`orElseGet()` also returns a default value when the `Optional` is empty.

The difference is that it accepts a **Supplier**, so the default value is created **only when needed**.

```java
String result =
        language.orElseGet(() -> "Java");
```

Programming model:

```text
orElseGet()
        │
expects
Supplier
        │
implemented using
Lambda Expression
or
Method Reference
```

Use `orElseGet()` when creating the default value is expensive, such as reading from a database or calling another service.

---

# orElse() vs orElseGet()

This is a very common interview question.

| `orElse()`                           | `orElseGet()`                           |
| ------------------------------------ | --------------------------------------- |
| Takes a value                        | Takes a `Supplier`                      |
| Default value is created immediately | Default value is created only if needed |
| Suitable for simple constants        | Suitable for expensive operations       |

Example:

```java
employee.orElse(createDefaultEmployee());
```

`createDefaultEmployee()` is executed even if the `Optional` contains a value.

```java
employee.orElseGet(
        () -> createDefaultEmployee()
);
```

Here, `createDefaultEmployee()` is executed **only if** the `Optional` is empty.

---

# orElseThrow()

`orElseThrow()` throws an exception if the value is absent.

```java
Employee employee =
        optional.orElseThrow(
                () -> new RuntimeException(
                        "Employee not found"
                )
        );
```

Programming model:

```text
orElseThrow()
        │
expects
Supplier
        │
returns Exception
```

Use `orElseThrow()` when the value is mandatory.

Examples:

- Employee must exist.

- Order must exist.

- Product must exist.

---

# map()

`map()` transforms the value inside an `Optional`.

It expects a **Function**.

```java
Optional<Employee> employee =
        findEmployee(101);

Optional<String> name =
        employee.map(Employee::getName);
```

Programming model:

```text
Optional<Employee>
        │
      map()
        │
        ▼
Optional<String>
```

If the `Optional` is empty, `map()` simply returns an empty `Optional`.

---

# flatMap()

`flatMap()` is similar to `map()`, but it avoids creating nested `Optional` objects.

Suppose:

```java
Optional<Employee>
```

and

```java
Employee::getManager
```

returns:

```java
Optional<Employee>
```

Using `map()`:

```java
Optional<Optional<Employee>>
```

Using `flatMap()`:

```java
Optional<Employee>
```

Programming model:

```text
map()
        │
Optional<Optional<T>>

flatMap()
        │
Optional<T>
```

Just like Stream's `flatMap()`, `Optional.flatMap()` removes one level of wrapping.

---

# map() vs flatMap()

| `map()`                         | `flatMap()`                                        |
| ------------------------------- | -------------------------------------------------- |
| Returns `Optional<T>`           | Returns flattened `Optional<T>`                    |
| Can create nested Optionals     | Avoids nested Optionals                            |
| Used for simple transformations | Used when the mapper already returns an `Optional` |

---

# Best Practices

- Use `Optional` as a **return type** when a value may be absent.

- Prefer `orElseGet()` over `orElse()` for expensive default values.

- Prefer `orElseThrow()` when a value is mandatory.

- Use `map()` to transform values inside an `Optional`.

- Use `flatMap()` when the transformation already returns an `Optional`.

---

# Common Mistakes

- Returning `null` instead of `Optional.empty()`.

- Calling `get()` without checking whether a value is present.

- Using `isPresent()` followed by `get()` instead of `ifPresent()`, `map()`, or `orElse()`.

- Using `Optional` for fields, method parameters, or entity classes. It is primarily intended as a **return type**.

- Using `orElse()` when creating the default value is expensive.

---d using Optional for:• Entity fields (e.g., JPA entities)• DTO fields• Method parameters• Serialization modelsUse Optional primarily as a method return type.
