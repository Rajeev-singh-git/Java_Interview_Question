# 07.2 Stream Operations

A Stream processes data by applying a series of operations called a **Stream Pipeline**.

These operations are broadly classified into:

* **Intermediate Operations**
* **Terminal Operations**

Understanding the difference between them is essential because every Stream pipeline follows this execution model.

---

# Stream Pipeline

```java
Source
(Collection, Array...)
        │
        ▼
Create Stream
        │
        ▼
Intermediate Operations
(filter, map, sorted...)
        │
        ▼
Terminal Operation
(collect, count, reduce...)
        │
        ▼
Result
```

Example:

```java
List<String> names =
        List.of("java", "spring", "docker");

List<String> result =
        names.stream()
             .filter(name -> name.length() > 4)
             .map(String::toUpperCase)
             .toList();
```

Pipeline:

```text
Source
(List)

↓

filter()

↓

map()

↓

toList()
```

---

# Intermediate vs Terminal Operations

## Intermediate Operations

Intermediate Operations transform the Stream and return **another Stream**.

They are **lazy**, meaning they do not execute immediately.

Examples:

* `filter()`
* `map()`
* `flatMap()`
* `distinct()`
* `sorted()`
* `limit()`
* `skip()`
* `peek()`

---

## Terminal Operations

Terminal Operations produce the final result or perform an action.

Once a Terminal Operation executes, the Stream is consumed and cannot be reused.

Examples:

* `collect()`
* `forEach()`
* `count()`
* `reduce()`
* `findFirst()`
* `findAny()`
* `anyMatch()`
* `allMatch()`
* `noneMatch()`

---

# Intermediate vs Terminal Operations

| Intermediate Operation | Terminal Operation    |
| ---------------------- | --------------------- |
| Returns another Stream | Produces final result |
| Lazy                   | Executes the pipeline |
| Can be chained         | Ends the pipeline     |
| Stream remains usable  | Stream is consumed    |

# filter()

The `filter()` operation selects elements that satisfy a condition.

It expects a **Predicate** as its parameter.

```java
List<Integer> numbers =
        List.of(2, 5, 8, 11, 14);

numbers.stream()
       .filter(n -> n % 2 == 0)
       .forEach(System.out::println);
```

Output

```text
2
8
14
```

Programming model:

```text
filter()
        │
expects
Predicate
        │
implemented using
Lambda Expression
```

---

# map()

The `map()` operation transforms each element into another value.

It expects a **Function** as its parameter.

```java
List<String> names =
        List.of("java", "spring", "docker");

names.stream()
     .map(String::toUpperCase)
     .forEach(System.out::println);
```

Output

```java
JAVA
SPRING
DOCKER
```

Programming model:

```text
map()
      │
expects
Function
      │
implemented using
Method Reference
or
Lambda Expression
```

Common use cases:

* Convert names to uppercase
* Convert entities to DTOs
* Calculate salaries
* Extract object properties

---

# filter() vs map()

This is one of the most commonly asked interview questions.

| filter()                        | map()                               |
| ------------------------------- | ----------------------------------- |
| Selects elements                | Transforms elements                 |
| Uses `Predicate`                | Uses `Function`                     |
| Number of elements may decrease | Number of elements remains the same |
| Returns matching elements       | Returns transformed elements        |

Example:

```java
numbers.stream()
       .filter(n -> n > 10);
```

Returns only numbers greater than 10.

```java
names.stream()
     .map(String::length);
```

Returns the length of every string.

---

# flatMap()

`flatMap()` transforms each element into a Stream and then **flattens** all those Streams into a single Stream.

It is commonly used when dealing with nested collections.

Suppose we have:

```java
List<List<String>> names =
        List.of(
                List.of("Java", "Spring"),
                List.of("Docker", "Kafka")
        );
```

Using `flatMap()`:

```java
names.stream()
     .flatMap(List::stream)
     .forEach(System.out::println);
```

Output

```text
Java
Spring
Docker
Kafka
```

Without `flatMap()`, the Stream would contain two `List<String>` objects instead of four individual strings.

Programming model:

```java
Nested Collections
        │
        ▼
     flatMap()
        │
        ▼
  Single Stream
```

Common use cases:

* Nested Lists
* Orders containing multiple items
* Departments containing multiple employees
* Reading multiple files

```java
Why `flatMap()` is important

A lot of developers memorize:

```java
flatMap(List::stream)
```

without understanding **why** it exists.

The key idea is:

* **`map()` → One element becomes one element.**
* **`flatMap()` → One element becomes many elements, and all results are flattened into a single Stream.**

That mental model is far more valuable than memorizing syntax, and it's exactly what interviewers usually test when they ask about `flatMap()`.

# distinct()

The `distinct()` operation removes duplicate elements from a Stream.

```java
List<Integer> numbers =
        List.of(2, 5, 2, 8, 5, 11);

numbers.stream()
       .distinct()
       .forEach(System.out::println);
```

Output

```text
2
5
8
11
```

Common use cases:

* Remove duplicate IDs
* Remove duplicate emails
* Remove repeated values

> **Note:** `distinct()` uses `equals()` and `hashCode()` to determine whether two objects are duplicates.

---

# sorted()

The `sorted()` operation sorts the elements in a Stream.

### Natural Ordering

```java
List<Integer> numbers =
        List.of(5, 2, 8, 1);

numbers.stream()
       .sorted()
       .forEach(System.out::println);
```

Output

```text
1
2
5
8
```

---

### Custom Sorting

```java
List<String> names =
        List.of("Java", "Spring", "Docker");

names.stream()
     .sorted(Comparator.comparing(String::length))
     .forEach(System.out::println);
```

Common use cases:

* Sort employees by salary
* Sort students by marks
* Sort products by price

---

# limit()

The `limit()` operation returns only the first **N** elements.

```java
List<Integer> numbers =
        List.of(2, 5, 8, 11, 14);

numbers.stream()
       .limit(3)
       .forEach(System.out::println);
```

Output

```text
2
5
8
```

Common use cases:

* Top 10 employees
* First 5 records
* Pagination

---

# skip()

The `skip()` operation ignores the first **N** elements.

```java
List<Integer> numbers =
        List.of(2, 5, 8, 11, 14);

numbers.stream()
       .skip(2)
       .forEach(System.out::println);
```

Output

```text
8
11
14
```

Common use cases:

* Pagination
* Ignore header records
* Skip already processed data

---

# limit() vs skip()

These operations are often used together.

```java
numbers.stream()
       .skip(10)
       .limit(5);
```

Programming model:

```text
Skip first 10 elements
            │
            ▼
Return next 5 elements
```

This is commonly used for implementing pagination.

---

# peek()

`peek()` performs an action on each element **without modifying** the Stream.

It is primarily intended for **debugging**.

```java
List<Integer> numbers =
        List.of(2, 5, 8);

numbers.stream()
       .peek(System.out::println)
       .map(n -> n * 2)
       .toList();
```

Output

```text
2
5
8
```

The values are printed before they are transformed.

---

## Real-world Usage

Suppose you're debugging a Stream pipeline.

```java
employees.stream()
         .filter(Employee::isActive)
         .peek(System.out::println)
         .map(Employee::getName)
         .toList();
```

`peek()` allows you to inspect the elements flowing through the pipeline without changing them.

---

# map() vs peek()

Another common interview question.

| map()                        | peek()                    |
| ---------------------------- | ------------------------- |
| Transforms elements          | Performs an action        |
| Returns modified elements    | Returns the same elements |
| Uses `Function`              | Uses `Consumer`           |
| Used for data transformation | Mainly used for debugging |

Example:

```java
numbers.stream()
       .map(n -> n * 2);
```

Every element changes.

```java
numbers.stream()
       .peek(System.out::println);
```

Elements remain unchanged.

| Operation    | Functional Interface | Purpose                 |
| ------------ | -------------------- | ----------------------- |
| `filter()`   | `Predicate`          | Select elements         |
| `map()`      | `Function`           | Transform elements      |
| `flatMap()`  | `Function`           | Flatten nested Streams  |
| `distinct()` | —                    | Remove duplicates       |
| `sorted()`   | `Comparator`         | Sort elements           |
| `limit()`    | —                    | Return first N elements |
| `skip()`     | —                    | Skip first N elements   |
| `peek()`     | `Consumer`           | Inspect elements        |

# forEach()

The `forEach()` operation performs an action on each element of the Stream.

It expects a **Consumer** as its parameter.

```java
List<String> names =
        List.of("Java", "Spring", "Docker");

names.stream()
     .forEach(System.out::println);
```

Programming model:

```java
forEach()
      │
   expects
  Consumer
      │
implemented  using
 Method Reference
       or
 Lambda Expression
```

Common use cases:

- Logging

- Printing results

- Sending notifications

- Processing each element

---

# collect()

The `collect()` operation converts a Stream into another form such as a `List`, `Set`, or `Map`.

It is one of the most frequently used terminal operations.

```java
List<String> result =
        names.stream()
             .map(String::toUpperCase)
             .toList();
```

or

```java
List<String> result =
        names.stream()
             .map(String::toUpperCase)
             .collect(Collectors.toList());
```

> **Note:** Java 16 introduced `toList()`. Prior to Java 16, `Collectors.toList()` was commonly used.

Common use cases:

- Convert Stream to List

- Convert Stream to Set

- Group data

- Aggregate results

> **Note:** `Collectors` is covered in detail in the next chapter.

---

# reduce()

The `reduce()` operation combines multiple elements into a **single result**.

```java
List<Integer> numbers =
        List.of(2, 5, 8, 11);

int sum =
        numbers.stream()
               .reduce(0, Integer::sum);

System.out.println(sum);
```

Output

```text
26
```

Common use cases:

- Sum

- Product

- Maximum value

- Minimum value

---

# count()

`count()` returns the number of elements in a Stream.

```java
long total =
        numbers.stream()
               .count();
```

Output

```text
4
```

---

# findFirst()

Returns the first element of the Stream.

```java
Optional<Integer> number =
        numbers.stream()
               .findFirst();
```

Since a Stream may be empty, it returns an `Optional`.

---

# findAny()

Returns any element from the Stream.

```java
Optional<Integer> number =
        numbers.stream()
               .findAny();
```

In sequential Streams, it often returns the first element.

In Parallel Streams, it may return any matching element for better performance.

---

# anyMatch()

Returns `true` if **at least one** element satisfies the condition.

```java
boolean result =
        numbers.stream()
               .anyMatch(n -> n > 10);
```

---

# allMatch()

Returns `true` if **every** element satisfies the condition.

```java
boolean result =
        numbers.stream()
               .allMatch(n -> n > 0);
```

---

# noneMatch()

Returns `true` if **no** element satisfies the condition.

```java
boolean result =
        numbers.stream()
               .noneMatch(n -> n < 0);
```

---

# findFirst() vs findAny()

| findFirst()                           | findAny()                       |
| ------------------------------------- | ------------------------------- |
| Returns the first element             | Returns any element             |
| Predictable                           | May return any matching element |
| Commonly used with sequential Streams | Optimized for Parallel Streams  |

---

# anyMatch() vs allMatch() vs noneMatch()

| Operation     | Returns `true` when...       |
| ------------- | ---------------------------- |
| `anyMatch()`  | At least one element matches |
| `allMatch()`  | Every element matches        |
| `noneMatch()` | No element matches           |

---

# Terminal Operation Summary

| Operation     | Functional Interface | Purpose                          |
| ------------- | -------------------- | -------------------------------- |
| `forEach()`   | `Consumer`           | Perform an action                |
| `collect()`   | Collector            | Convert Stream into another form |
| `reduce()`    | `BinaryOperator`     | Combine elements                 |
| `count()`     | —                    | Count elements                   |
| `findFirst()` | —                    | Return first element             |
| `findAny()`   | —                    | Return any element               |
| `anyMatch()`  | `Predicate`          | At least one matches             |
| `allMatch()`  | `Predicate`          | Every element matches            |
| `noneMatch()` | `Predicate`          | No element matches               |

---

# Best Practices

- Keep Stream pipelines short and readable.

- Prefer Method References when they improve readability.

- Use `map()` for transformation and `filter()` for filtering.

- Use `peek()` only for debugging.

- Avoid modifying external state inside Stream operations.

- Use `findFirst()` when order matters and `findAny()` when it does not.

---

# Common Mistakes

- Forgetting that Streams can be consumed only once.

- Using `map()` when `filter()` is required.

- Using `peek()` to modify data instead of debugging.

- Creating overly complex Stream pipelines that hurt readability.

- Forgetting that intermediate operations are lazy.
