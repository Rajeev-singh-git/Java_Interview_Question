# 09. Parallel Streams

A **Parallel Stream** is a Stream that processes data using **multiple threads** instead of a single thread.

Unlike a normal Stream, which processes elements one after another, a Parallel Stream divides the work into smaller tasks and executes them concurrently, potentially improving performance for suitable workloads.

---

# Why were Parallel Streams Introduced?

A normal (Sequential) Stream processes elements one at a time.

```java
numbers.stream()
       .filter(n -> n % 2 == 0)
       .map(n -> n * 2)
       .toList();
```

Processing Flow

```text
Element 1

↓

Element 2

↓

Element 3

↓

...
```

Only one thread processes the entire pipeline.

For large datasets and CPU-intensive operations, this may not fully utilize modern multi-core processors.

To address this, Java 8 introduced **Parallel Streams**, allowing work to be distributed across multiple CPU cores automatically.

---

# Sequential Stream vs Parallel Stream

## Sequential Stream

```java
numbers.stream()
       .forEach(System.out::println);
```

Processes elements one after another.

---

## Parallel Stream

```java
numbers.parallelStream()
       .forEach(System.out::println);
```

Processes multiple elements simultaneously using multiple threads.

---

# Sequential vs Parallel Streams

| Sequential Stream             | Parallel Stream                            |
| ----------------------------- | ------------------------------------------ |
| Uses one thread               | Uses multiple threads                      |
| Processes elements one by one | Processes multiple elements simultaneously |
| Predictable execution order   | Execution order may vary                   |
| Lower overhead                | Additional thread management overhead      |
| Suitable for small datasets   | Suitable for large CPU-intensive workloads |

---

# How Parallel Streams Work

One of the biggest advantages of Parallel Streams is that developers **do not manage threads manually**.

When a Parallel Stream is created:

1. The data is divided into smaller tasks.

2. The tasks are distributed across multiple threads.

3. Each thread processes its assigned data.

4. The results are combined automatically.

```java
Large Collection
        │
        ▼
Split into Multiple Tasks
        │
 ┌──────┼──────┐
 ▼      ▼      ▼
Task1  Task2  Task3
 │       │      │
 ▼       ▼      ▼
Worker Threads
        │
        ▼
Combine Results
```

The entire process is managed internally by the Stream API.

---

# Fork/Join Framework

Parallel Streams are built on top of Java's **Fork/Join Framework**.

The Fork/Join Framework follows a simple strategy:

```java
Large Task

↓

Fork
(Split into smaller tasks)

↓

Execute in Parallel

↓

Join
(Combine results)
```

Developers typically do **not** interact with the Fork/Join Framework directly when using Parallel Streams.

The Stream API handles task splitting, scheduling, and result merging automatically.

---

# Creating Parallel Streams

## Using `parallelStream()`

The most common approach.

```java
List<Integer> numbers =
        List.of(1, 2, 3, 4, 5);

numbers.parallelStream()
       .forEach(System.out::println);
```

---

## Using `parallel()`

Convert an existing Stream into a Parallel Stream.

```java
numbers.stream()
       .parallel()
       .forEach(System.out::println);
```

Both approaches produce a Parallel Stream.

---

# parallelStream() vs parallel()

| `parallelStream()`                                   | `parallel()`                                       |
| ---------------------------------------------------- | -------------------------------------------------- |
| Creates a Parallel Stream directly from a Collection | Converts an existing Stream into a Parallel Stream |
| Called on a Collection                               | Called on a Stream                                 |

---

# Real-world Usage

Parallel Streams are useful for **CPU-intensive operations** on large datasets.

Examples:

- Image processing

- Financial calculations

- Report generation

- Scientific computations

- Large-scale data analysis

They are generally **not** intended for database calls, REST API calls, or other I/O-bound operations.

---

# Key Points to Remember

- A Parallel Stream processes data using multiple threads.

- It is built on the Fork/Join Framework.

- Developers do not manage thread creation manually.

- Parallel processing is automatic.

- Parallel Streams are most beneficial for large CPU-intensive tasks.

---

# When Should You Use Parallel Streams?

Parallel Streams are **not always faster**.

They provide benefits only when the workload can be efficiently divided across multiple CPU cores.

Parallel Streams are a good choice when:

- Processing a large dataset.

- Operations are CPU-intensive.

- Individual operations are independent.

- The workload can be parallelized easily.

Examples:

- Image processing

- Data analytics

- Mathematical computations

- Large report generation

---

# When Should You Avoid Parallel Streams?

Avoid Parallel Streams when:

- Processing a small collection.

- Performing database operations.

- Calling REST APIs.

- Reading or writing files.

- Operations depend on execution order.

- Using shared mutable state.

In these situations, the overhead of thread management often outweighs the benefits.

---

# Performance Considerations

Many developers assume:

> **Parallel Stream = Faster**

This is **not true**.

Performance depends on several factors:

- Dataset size

- Number of CPU cores

- Cost of each operation

- Thread management overhead

- Splitting and merging costs

For small collections, a Sequential Stream is often faster.

---

# Thread Safety

Parallel Streams execute operations on multiple threads.

This means modifying shared mutable data can produce incorrect results.

❌ Incorrect

```java
List<Integer> result =
        new ArrayList<>();

numbers.parallelStream()
       .forEach(result::add);
```

Multiple threads modify the same `ArrayList`, leading to unpredictable behavior.

---

✅ Better

```java
List<Integer> result =
        numbers.parallelStream()
               .filter(n -> n % 2 == 0)
               .toList();
```

Each thread processes its own data, and the Stream framework safely combines the results.

---

# Ordering

A Sequential Stream preserves encounter order.

```java
numbers.stream()
       .forEach(System.out::println);
```

Output

```text
1
2
3
4
5
```

---

A Parallel Stream does **not** guarantee the processing order.

```java
numbers.parallelStream()
       .forEach(System.out::println);
```

Possible Output

```text
3
1
5
2
4
```

---

# forEach() vs forEachOrdered()

If order matters, use `forEachOrdered()`.

```java
numbers.parallelStream()
       .forEachOrdered(System.out::println);
```

| `forEach()`                 | `forEachOrdered()`        |
| --------------------------- | ------------------------- |
| Faster                      | Slightly slower           |
| Order not guaranteed        | Preserves encounter order |
| Best for maximum throughput | Best when order matters   |

---

# Parallel Streams vs Traditional Multithreading

| Parallel Streams             | Traditional Multithreading                  |
| ---------------------------- | ------------------------------------------- |
| High-level API               | Low-level thread management                 |
| Automatic thread handling    | Manual thread creation                      |
| Built on Fork/Join Framework | Uses `Thread`, `ExecutorService`, etc.      |
| Suitable for data processing | Suitable for general concurrent programming |

---

# Best Practices

- Use Parallel Streams only after measuring performance.

- Prefer Sequential Streams for small collections.

- Avoid shared mutable state.

- Use stateless Lambda Expressions.

- Use `forEachOrdered()` only when ordering is required.

- Keep stream operations independent.

---

# Common Mistakes

- Assuming Parallel Streams are always faster.

- Using Parallel Streams for database or REST API calls.

- Modifying shared collections inside `forEach()`.

- Ignoring ordering requirements.

- Using Parallel Streams without performance testing.
