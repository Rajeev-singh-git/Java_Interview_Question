# 10. Java 8 Best Practices & Performance

This chapter summarizes the most important best practices, performance considerations, and common pitfalls when using Java 8 features in production applications.

---

## Lambda Expressions

### ✅ Prefer Lambda Expressions for short behavior.

```java
list.forEach(System.out::println);
```

---

### ❌ Avoid long Lambda Expressions.

If a Lambda contains complex business logic, extract it into a separate method.

---

## Method References

### ✅ Prefer Method References when they improve readability.

```java
.map(String::toUpperCase)
```

instead of

```java
.map(name -> name.toUpperCase())
```

---

### ❌ Don't force Method References.

Use them only when they make the code simpler.

---

## Functional Interfaces

### ✅ Use the most appropriate Functional Interface.

| Requirement      | Use       |
| ---------------- | --------- |
| Condition        | Predicate |
| Transformation   | Function  |
| Action           | Consumer  |
| Value Generation | Supplier  |

---

### ❌ Don't create custom Functional Interfaces unnecessarily.

Prefer Java's built-in Functional Interfaces.

---

## Streams

### ✅ Prefer Streams for collection processing.

```java
employees.stream()
         .filter(Employee::isActive)
         .toList();
```

---

### ❌ Don't use Streams for everything.

Sometimes a simple `for` loop is more readable.

---

### ✅ Keep Stream pipelines short.

Good

```java
filter()

↓

map()

↓

collect()
```

Avoid very long pipelines that are difficult to read.

---

### ✅ Prefer Method References inside Streams.

```java
.map(Employee::getName)
```

---

### ❌ Avoid side effects inside Streams.

Bad

```java
list.stream()
    .forEach(result::add);
```

Prefer collecting results instead.

---

## Collectors

### ✅ Use the Collector that best matches the problem.

| Requirement  | Collector        |
| ------------ | ---------------- |
| Group        | groupingBy()     |
| Partition    | partitioningBy() |
| Statistics   | summarizingInt() |
| Join Strings | joining()        |

---

### ❌ Avoid multiple traversals.

Prefer

```java
summarizingInt()
```

instead of multiple Stream operations.

---

## Optional

### ✅ Use Optional as a return type.

```java
Optional<Employee>
```

---

### ❌ Don't use Optional for:

* Entity fields
* DTO fields
* Method parameters

---

### ✅ Prefer expressive APIs.

Good

```java
orElseGet()

map()

orElseThrow()
```

Avoid

```java
isPresent()

↓

get()
```

---

## Parallel Streams

### ✅ Use Parallel Streams only for:

* Large datasets
* CPU-intensive tasks
* Independent operations

---

### ❌ Avoid Parallel Streams for:

* Database calls
* REST API calls
* File I/O
* Small collections

---

### ❌ Never assume Parallel Streams are faster.

Always measure performance.

---

## Performance Tips

* Prefer primitive Streams (`IntStream`, `LongStream`, `DoubleStream`) for numeric computations.
* Avoid unnecessary object creation inside Stream pipelines.
* Keep Lambda Expressions stateless.
* Prefer lazy Stream operations over manual temporary collections.
* Measure performance before optimizing.

---

## Common Mistakes

* Writing overly complex Stream pipelines.
* Using `peek()` for business logic.
* Reusing consumed Streams.
* Calling `Optional.get()` without checking.
* Using `ParallelStream` without understanding the workload.
* Choosing Streams when a loop is simpler.

---

## Production Checklist

Before writing Java 8 code, ask yourself:

* Is a Stream clearer than a loop?
* Am I using the correct Functional Interface?
* Is this Lambda readable?
* Can a Method Reference improve readability?
* Am I using the right Collector?
* Should this method return an `Optional`?
* Is a Parallel Stream actually beneficial?
* Am I optimizing based on measurements rather than assumptions?

---

## Final Takeaway

> **Java 8 features should improve readability, maintainability, and correctness—not simply make the code more "functional." Choose the simplest solution that clearly expresses the intent while balancing performance and maintainability.**
